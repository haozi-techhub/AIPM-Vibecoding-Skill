# 图表生成指南

本文档指导如何在PRD中生成各类图表，包括Mermaid语法示例和lark-whiteboard使用说明。

**核心原则**：图表的目的是降低理解成本，不是展示复杂性。清晰 > 全面 > 美观。

---

## 1. 产品架构图（三层）

### 宏观层：系统定位图

展示产品在整个技术生态中的位置和外部依赖关系。

**Mermaid示例**：

```mermaid
graph TB
    subgraph 外部系统
        A[用户终端] 
        B[第三方API]
        C[数据平台]
    end
    
    subgraph 本产品
        D[接入层]
        E[业务层]
        F[数据层]
    end
    
    subgraph 基础设施
        G[云服务]
        H[监控系统]
    end
    
    A --> D
    B --> E
    E --> F
    F --> C
    E --> G
    D --> H
```

**写作要点**：
- 只展示系统级别的关系，不要进入模块细节
- 外部系统用不同颜色/区域区分
- 标注关键的数据流方向

### 中观层：模块关系图

展示核心模块间的交互关系和数据流转。

**Mermaid示例**：

```mermaid
graph LR
    subgraph 前端模块
        A[对话界面]
        B[设置面板]
        C[历史记录]
    end
    
    subgraph 核心服务
        D[会话管理]
        E[意图识别]
        F[内容生成]
        G[知识检索]
    end
    
    subgraph 基础服务
        H[用户服务]
        I[存储服务]
        J[模型网关]
    end
    
    A -->|用户输入| D
    D -->|分析请求| E
    E -->|生成任务| F
    E -->|检索任务| G
    F -->|调用模型| J
    G -->|查询| I
    D -->|用户信息| H
    C -->|读取| I
```

**写作要点**：
- 模块之间的连线标注数据类型或调用目的
- 按前端/核心/基础分层排列
- 突出核心服务之间的关系

### 微观层：模块内部结构图

展示关键模块内部的组件关系。

**Mermaid示例**：

```mermaid
graph TB
    subgraph 内容生成模块
        A[请求解析器]
        B[Prompt构造器]
        C[模型路由器]
        D[响应后处理]
        E[质量检测器]
        F[缓存管理器]
    end
    
    A -->|结构化请求| B
    B -->|完整Prompt| C
    C -->|模型输出| D
    D -->|生成结果| E
    E -->|通过| F
    E -->|不通过| B
    F -->|缓存| A
```

**写作要点**：
- 只对1-2个最关键的模块做微观展开
- 标注组件间的数据转换
- 包含错误/重试路径

## 2. 用户流程图（决策树风格）

### 核心特征

用户流程图采用「用户决策 + 系统响应」双轨设计，让读者同时理解用户行为和系统行为。

**Mermaid示例**：

```mermaid
flowchart TD
    Start([用户打开应用]) --> A{是否登录？}
    A -->|已登录| B[展示个性化首页]
    A -->|未登录| C[展示引导页]
    
    C --> D{用户选择}
    D -->|注册| E[注册流程]
    D -->|跳过| F[展示基础首页]
    D -->|登录| G[登录流程]
    
    B --> H{用户意图}
    H -->|创建内容| I[进入创作流程]
    H -->|浏览内容| J[进入浏览流程]
    H -->|管理设置| K[进入设置流程]
    
    I --> L{选择创作类型}
    L -->|文本| M[文本编辑器]
    L -->|图片| N[图片生成器]
    L -->|多模态| O[混合编辑器]
    
    M --> P{内容满意？}
    P -->|满意| Q[发布/保存]
    P -->|不满意| R[AI优化建议]
    R --> M
```

**写作要点**：
- 菱形框表示用户决策点，矩形框表示系统动作
- 每个决策点标注所有可能的选项
- 标注循环路径（如"不满意→AI优化→重新编辑"）
- 所有路径都要有明确终点

### 异常流程补充

主流程图外，单独绘制异常处理流程：

```mermaid
flowchart TD
    A[用户操作] --> B{系统检查}
    B -->|正常| C[正常流程继续]
    B -->|网络异常| D[展示离线提示]
    B -->|权限不足| E[引导开通权限]
    B -->|内容违规| F[展示违规提示]
    
    D --> G{网络恢复？}
    G -->|是| H[自动重试]
    G -->|否| I[保存草稿到本地]
    
    E --> J{用户选择}
    J -->|同意开通| K[跳转开通页]
    J -->|拒绝| L[降级展示基础功能]
```

## 3. 交互流程与关键帧

### 关键帧序列图

展示一个完整场景中的关键交互节点：

```mermaid
sequenceDiagram
    participant U as 用户
    participant UI as 界面
    participant S as 系统
    participant AI as AI模型
    
    U->>UI: 输入创作需求
    UI->>S: 解析用户意图
    S->>AI: 发送生成请求
    
    Note over UI: 关键帧1：加载态展示
    AI-->>S: 流式返回结果
    S-->>UI: 实时渲染内容
    
    Note over UI: 关键帧2：内容生成中
    U->>UI: 实时编辑/调整
    UI->>S: 更新生成参数
    
    Note over UI: 关键帧3：编辑态
    U->>UI: 确认满意，点击保存
    UI->>S: 保存最终版本
    S-->>UI: 保存成功反馈
    
    Note over UI: 关键帧4：完成态
```

**写作要点**：
- Note标注是关键帧的位置
- 实线表示主动请求，虚线表示响应
- 标注关键的等待/加载时间点

## 4. 指标树图

### 北极星指标拆解

```mermaid
graph TD
    A[北极星指标<br/>月活跃创作用户数]
    
    A --> B[获取效率<br/>新用户注册转化率]
    A --> C[活跃深度<br/>周均创作次数]
    A --> D[留存质量<br/>次月留存率]
    
    B --> B1[渠道触达量]
    B --> B2[落地页转化率]
    B --> B3[注册完成率]
    
    C --> C1[首次创作引导完成率]
    C --> C2[创作功能使用率]
    C --> C3[AI辅助采纳率]
    
    D --> D1[内容质量满意度]
    D --> D2[功能使用频次]
    D --> D3[社交分享率]
    
    A --> E[反向指标]
    E --> E1[用户投诉率 < 0.1%]
    E --> E2[内容违规率 < 0.01%]
    E --> E3[系统错误率 < 0.5%]
    
    style A fill:#FF6B35,color:#fff
    style E fill:#FF4444,color:#fff
```

**写作要点**：
- 北极星指标用醒目颜色标注
- 反向指标用红色系标注
- 每个叶子节点标注目标值
- 层级不超过3层

## 5. 评测维度图

### 评测体系全景

```mermaid
graph LR
    subgraph 功能评测
        A1[功能正确性]
        A2[边界case覆盖]
        A3[兼容性测试]
    end
    
    subgraph 体验评测
        B1[响应速度]
        B2[交互流畅度]
        B3[视觉一致性]
    end
    
    subgraph AI评测
        C1[生成质量]
        C2[幻觉率]
        C3[安全合规]
    end
    
    subgraph 数据评测
        D1[埋点覆盖率]
        D2[数据准确性]
        D3[指标可观测性]
    end
    
    A1 --> E[综合评分]
    B1 --> E
    C1 --> E
    D1 --> E
    
    E --> F{是否达标？}
    F -->|是| G[允许上线]
    F -->|否| H[阻断上线并修复]
```

## 6. 使用lark-whiteboard绘制

### 适用场景

当以下情况时，优先使用lark-whiteboard：
- 需要更灵活的布局（Mermaid不支持自由定位）
- 需要嵌入到飞书文档中
- 需要添加颜色编码、图标等视觉元素
- 团队更习惯在飞书中查看

### 调用方式

通过lark-whiteboard skill绘制，然后嵌入飞书文档。典型用法：

1. 明确图表类型（架构图/流程图/思维导图/时序图）
2. 准备图表数据结构
3. 调用lark-whiteboard生成
4. 获取图表链接嵌入PRD文档

### 降级方案

如果lark-whiteboard不可用，使用Mermaid代码块作为降级：
- 在飞书文档中以代码块形式嵌入
- 添加文字说明辅助理解
- 后续手动替换为正式图表

## 图表通用原则

1. **标注所有决策点**：每个菱形（决策）必须标注条件和所有分支
2. **清晰 > 全面**：一张图传达一个核心信息，不要试图在一张图里说完所有事
3. **统一配色**：同类元素使用同一颜色系，不超过5种主色
4. **文字精简**：图中文字不超过10个字，详细说明放在图外
5. **双向流转**：如果存在循环/重试，必须标注退出条件
6. **读者导向**：开发看的图强调数据流，PM看的图强调决策逻辑，设计看的图强调交互帧
