# 面试录音复盘 Skill

**CRITICAL — 开始前 MUST 先用 Read 工具读取 [`../lark-shared/SKILL.md`](../lark-shared/SKILL.md)，其中包含飞书认证和权限处理**

从面试录音转录文本出发，产出结构化复盘文档，帮助用户真正提升面试能力（不是背答案）。

## 核心原则

1. **教学优于话术** — Part 2「正确答案」不是给一段更好的话术让用户背，而是真正教会知识点的本质。用户背不下AI写的话术，但理解了知识后能用自己的话回答。
2. **项目文档是 ground truth** — 生成「项目深挖清单」时，答案来自用户自己的项目文档（PRD、架构文档、评测SOP等），不是AI编造。文档没覆盖的如实标注"文档中未覆盖，需要你自己补充"。
3. **聚焦3-5题** — Part 2-4只对答得最差的3-5个关键问题做深度展开。全做会信息过载，用户抓不住重点。
4. **动态读取不缓存** — 简历、项目文档、Obsidian知识库都在持续更新。每次执行必须重新读取最新版本，不使用任何缓存。知识库采用两级查找：先查 `/Users/haozi/Documents/GitHub/obsidian-vault/原子笔记/`（轻量原子笔记，快速定位概念），需要更详细内容时再查 `/Users/haozi/Documents/GitHub/obsidian-vault/沉淀/`（完整方法论和深度文档）。
5. **发布前必须审核** — 所有生成内容在发布到飞书前，必须经过三重审核：可用性（用户能不能在真实面试中说出来）、真实性（是否符合用户实际经历）、知识库一致性（是否与用户已有知识体系吻合）。审核不通过的内容必须修正或标注警告后才能发布。
6. **以senior（年薪55万级）为对标** — 用户的目标是年薪55万+的AI PM岗位（P7+/P7强水平）。所有回答示例、训练任务、自查清单都以senior级标准设计，并明确标注用户当前的level评级（初级/中级/高级）和差距。**所有训练任务和自查项必须配套参考答案，不能只列任务不给答案**——清单没有答案 = 没有训练价值。
7. **可复用话术沉淀到话术库** — 每场复盘把"和你绑定、换家公司还能用"的问答和提问套路，追加进一份跨面试常驻的飞书「我的面试话术库」（见 Phase 6）。判断标准只问一句"换家公司这条还用得上吗"：用得上才收。库里保留面试官**原话**（不清洗），同一道题被多家问到就累积多条原话，形成"真实问法变体库"；行为题答案只打磨表达、不编造事实，标注待用户确认。

## 工作流

```
Phase 0: 输入收集
  转录文本 + 面试录音链接 + 目标JD + 讨论了哪些项目 + 公司名 + 日期 + 面试官角色
    ↓
Phase 1: 转录处理 → QA配对（Part 1）
  清洗口语 → 切分QA对（按面试官角色调整诊断重点） → 选出3-5个关键弱题
    ↓
Phase 2: 知识解析 → 正确答案教学 + senior对标（Part 2）
  [动态] 读Obsidian知识库 + 项目文档 + 简历
  每题输出: 正确理解 / 在你项目中 / 关键数据 / 行业实践
           + Senior（55万级）回答示例 + 你当前level评级
    ↓
Phase 3: 思维路径生成（Part 3）
  每个弱题建模 Step1→Step2→Step3→Step4 条件反射路径
    ↓
Phase 4: 追问防御链（Part 4）
  每个弱题延伸2-3个追问 + 意图 + 关键词
    ↓
Phase 5: 差距分析 + 训练闭环（Part 5）
  5a 项目深挖清单 | 5b 知识盲区清单 | 5c JD匹配映射
  5d 本周训练任务（每条配参考答案+验收标准）
  5e 下次面试前自查项（每条配标准答案+答案来源）
    ↓
Phase 6: 提炼可复用话术 → 追加话术库（跨面试常驻）
  Block 1 高频题×标准话术（含面试官原话集+你的成品答）| Block 2 面试官提问套路库
  产出：话术库增量 + 复盘文档里的增量摘要段
    ↓
Phase 7: 三重审核（发布前质量关卡）
  可用性审核 → 真实性审核 → 知识库一致性审核（含话术库增量）
  不通过的内容修正或标注警告
    ↓
Phase 8: 组装飞书文档 → lark-cli发布（复盘文档 + 追加话术库）→ 返回链接
```

## Phase 0：输入收集

需要收集6个输入。如果用户没有一次性提供全部，**用纯文本方式在对话中提问**，让用户自己输入回答。

**严禁使用 AskUserQuestion 工具弹窗选择**——用户希望保持对话流的自然性，自己用文本回答即可。直接在对话里列出问题清单，等用户回复。

### 输入清单（按询问顺序）

**询问顺序原则：先问短的元信息，最后问长的转录文本**——避免用户先粘贴一大段转录文本后还要继续被问其他问题。

| 顺序 | 输入 | 必填 | 接受格式 | 说明 |
|------|------|------|---------|------|
| 1 | 公司名 | 是 | 文本 | 用于文档标题 |
| 2 | 面试录音链接 | 推荐 | URL（钉钉/飞书/网盘录音回放链接） | 放在复盘文档最顶部，方便回听原音；只展示不转写，没有则省略 |
| 3 | 讨论的项目 | 是 | 项目名 / 飞书链接 / 本地文件路径 / 自由描述（任意组合，可多个） | 开放式询问，不预设选项 |
| 4 | 面试官角色 | 是 | HR / HM直属leader / 同级技术伙伴 / 高管(VP/CTO) / 不确定 | 决定Phase 1-4的诊断重点和追问预测 |
| 5 | 目标JD | 推荐 | 粘贴文本 / 图片路径 / URL | 没有则跳过Part 5c岗位匹配 |
| 6 | 面试日期 | 否 | YYYY-MM-DD | 默认当天 |
| 7 | 转录文本 | 是 | 粘贴文本 / 本地文件路径(.md/.txt) / 飞书文档链接 | **最后问**，来自钉钉转录或手动整理 |

### 面试官角色考察重点（用于Phase 1-4调整诊断重点）

| 面试官角色 | 考察重点 | 诊断侧重 | 追问类型 |
|-----------|---------|---------|---------|
| HR | 动机、稳定性、薪资预期、文化匹配 | 离职原因表达、职业规划逻辑 | 软性追问（家庭/异地/期待） |
| HM 直属leader | 执行力、专业能力、项目细节 | 项目深度、技术理解、协作经验 | 项目细节追问（trade-off/数据/决策） |
| 同级技术伙伴 | 技术深度、概念准确性、行业认知 | 知识准确度、术语规范、广度 | 概念追问（原理/对比/最新进展） |
| 高管 (VP/CTO) | 战略视野、判断力、行业格局 | 商业理解、宏观判断、组织能力 | 战略追问（为什么这个方向/如何规模化） |

### 项目询问（开放式，纯文本提问）

**不预设选项、不弹窗、不用AskUserQuestion**——直接在对话中用文本提问：

```
"这场面试中你聊到了哪些项目？请告诉我：
- 项目名称（如果有）
- 项目链接（飞书文档/本地文件路径/Obsidian笔记路径都行）
- 或者直接描述（项目背景、你的角色、做了什么）

可以提供一个或多个项目。"
```

等用户用文本回复后，按下方规则处理。

### 用户回复的处理逻辑

用户可能以以下任意方式回复，skill需要全部支持：

| 用户回复形式 | 处理方式 |
|------------|---------|
| 项目名称（如"小剧场"） | 在 `references/project-registry.md` 查找匹配项，找到则用注册表中的文档路径 |
| 飞书文档链接 | 用 `lark-cli docs +fetch --doc <url> --as user` 读取内容作为项目文档 |
| 本地文件路径 | 用 Read 工具读取（支持md/txt/pdf）|
| Obsidian笔记路径 | 用 Read 工具读取 |
| 项目目录路径 | 用 Bash 的 `ls` 列出目录中的md文件，逐个读取 |
| 自由文本描述 | 用用户描述作为项目背景信息，标注"无项目文档可参考" |
| 混合形式（项目名+链接+描述） | 全部接收，分别处理 |

### 处理规则

1. **优先使用注册表** — 如果用户提的项目名能在 `references/project-registry.md` 找到，自动加载该项目的全部文档（不需要用户重复提供）
2. **链接/路径直接读取** — 用户提供链接或路径时，直接读取内容，不要求用户再选
3. **未注册项目的兼容** — 如果是注册表外的新项目，用户提供的文档作为唯一参考；同时**建议用户更新 project-registry.md**以便后续复用
4. **多项目支持** — 用户可能讨论多个项目，全部读取后在 Part 5 项目深挖清单中分别处理

### 输入验证

- 转录文本至少500字符（真实面试远多于此）
- 如果是飞书链接，用 `lark-cli docs +fetch --doc <url>` 获取内容
- 如果JD是URL，用 WebFetch 获取；如果是图片，用 Read 工具读取
- 面试录音链接只做展示（嵌入文档顶部），**不要 fetch、不要下载、不要转写**——直接作为链接放进文档即可

## Phase 1：转录处理（输出 Part 1）

### Step 1.1：清洗口语

去除语气词（嗯、啊、那个、就是说）、重复表述、false starts。**保留候选人的原始语义**——清洗是为了可读性，不是重写。

### Step 1.2：识别角色

- 如果转录有说话人标签（如"发言人1/发言人2"），按标签区分
- 否则按提问/回答模式判断：提问的是面试官，回答的是候选人

### Step 1.3：切分QA对

每个QA对包含：
- **面试官问题**（清理口语后）
- **候选人实际回答**（保留原意）
- **简短诊断**（2-3句）：这个回答好在哪/差在哪，标注严重度

严重度标注：
- 🔴 **关键失分**：事实性错误、完全答不上、严重减分
- 🟡 **有待改进**：方向对但不够深、缺少数据支撑、表述不清
- 🟢 **表现不错**：回答到位，可能有小的改进空间

### Step 1.4：选出3-5个关键弱题

从所有QA对中选出答得最差的3-5个，用于Part 2-4深度展开。选择优先级：

```
事实性错误/完全答不上 > 面试官反复追问仍答不好 > 核心PM/AI高频题 > 直接对应JD要求的题
```

## Phase 2：知识解析 → 正确答案教学（输出 Part 2）

这是最核心的Phase。对每个选出的弱题，生成真正的知识教学内容。

### 知识解析策略（三层）

**Layer 1 — 问题分类**

将弱题映射到知识域。关键词识别：

| 关键词 | 知识域 | Obsidian目录 |
|--------|--------|-------------|
| Agent/多Agent/编排/orchestration | Agent架构 | AI-Agent框架/ |
| 评测/评估/指标/badcase | 评测体系 | AI-Agent框架/ + AI产品与评测/ |
| Prompt/提示词/系统提示 | Prompt工程 | AI产品与评测/ |
| RAG/检索/知识库/向量 | RAG | AI开发工作流/ |
| 竞品/对比/市场 | 竞品分析 | AI产品与评测/ |
| 需求/用户/场景/痛点 | 需求分析 | AI产品与评测/ |
| 具体项目名 | 项目专属 | 按project-registry.md |

**Layer 2 — 两级知识查找**

知识库采用**原子笔记优先、沉淀文档补充**的两级策略：

**第一级：原子笔记**（`/Users/haozi/Documents/GitHub/obsidian-vault/原子笔记/`）
- 先扫描原子笔记目录，按关键词匹配找到相关笔记
- 原子笔记是轻量级的概念卡片，用于快速确认用户是否学过某个概念、以及用户自己的理解方式

**第二级：沉淀文档**（`/Users/haozi/Documents/GitHub/obsidian-vault/沉淀/`）
- 当原子笔记内容不够深（缺少框架细节、缺少对比分析、缺少实操方法），再按 [`references/knowledge-vault-index.md`](references/knowledge-vault-index.md) 读取沉淀目录下的完整方法论文档
- 每个知识域先读标注 `(start here)` 的综合性文件

**补充规则：**
1. 项目相关问题额外读对应项目文档（从 project-registry.md 取路径）
2. **总读取文件控制在8-10个以内**，按与弱题的相关性排优先级
3. 始终读取简历（PDF）和STAR故事作为交叉参考

**Layer 3 — 交叉验证**

- 数据/指标：与简历和项目文档交叉核对，确保一致
- 知识框架：确认在用户vault中存在，不编造用户没学过的框架
- vault中没有的知识：用 WebSearch 补充，但明确标注"外部知识，尚未收入你的vault"

### 每题输出结构

对每个弱题，输出5个子节：

```
### 正确理解
这个问题的本质是什么？涉及什么概念/框架/原理？
（来源：Obsidian知识库）

### 在你的项目中
这个知识点在你的项目中具体是怎么回事？
（来源：项目文档，用项目文档中的真实细节）

### 关键数据解释
如果涉及指标/数据，怎么定义、怎么观测、怎么解释？
（来源：项目文档 + 简历，交叉核对）

### 行业最佳实践
业内领先团队怎么做的？
（来源：知识库 or WebSearch）

### Senior（55万级）回答示例 + 你的Level评级
这是核心模块——以senior标准对标用户当前回答。
```

### Senior级回答示例的写法

不是给一段话术让用户背，而是给**结构示范**。包含3个层次：

```
【senior级回答的结构】
开头（破题+定结论）：用一句话给出明确判断，让面试官立刻知道你的立场
中段（数据+论据）：用3-4个具体数据/事实支撑结论，每个数据带来源
收尾（战略视野）：跳出问题本身，谈这个决策对业务/产品/团队的更广泛影响
```

然后基于这个结构，给出**针对本题的senior级范例**：用用户真实项目的数据填充，写一段约150-250字的范例回答。

### Level评级标准

对用户在本题的实际回答，给出明确评级：

| Level | 表现特征 | 对应薪资区间 |
|-------|---------|------------|
| 🔴 初级 | 答错、答不上、答得空洞 | 25-35万 |
| 🟡 中级 | 答对方向但缺数据/缺trade-off/缺战略视野 | 35-45万 |
| 🟢 高级（senior） | 有结构、有数据、有战略、能hold住追问 | 45-60万（55万目标的target区间） |
| 🌟 专家级 | 能站在更高视角批判性思考，提出原创见解 | 60万+ |

评级时必须明确指出：
1. 用户当前在哪一档
2. 离senior级（55万target）还差什么具体维度
3. 这个gap可以通过什么训练补上

## Phase 3：思维路径生成（输出 Part 3）

对同样的3-5个弱题，生成条件反射训练路径。

### 格式

```plaintext {wrap}
听到"{面试官的问题}" →

Step 1（先定义/给结论）：{听到这个问题后，第一反应应该想什么}
Step 2（说原因/trade-off）：{为什么这么做/不这么做}
Step 3（对比其他方案）：{还有哪些方案，为什么不选}
Step 4（回到你的场景和结果）：{在你的项目中怎么做的，结果如何}
```

### 要求

- 每步用一句话描述，不写长段
- 目标是训练条件反射，不是提供完整答案
- 思维路径应参考知识库中的框架（比如Agent知识体系中的分类方式）
- Step 4必须映射到用户的真实项目经验

## Phase 4：追问防御链（输出 Part 4）

对同样的3-5个弱题，列出面试官最可能的追问。

### 每个追问包含

- **追问问题**（面试官可能怎么问）
- **追问意图**（他在测试什么能力/知识）
- **思路关键词**（不是完整答案，是你该想到的2-3个关键点）

### 追问来源

1. 面试中实际发生的追问（从转录中提取）
2. 按问题主题推导的高概率追问
3. 参考已有追问题库（如项目文档中的`面试追问题库.md`）

每个弱题列2-3个追问即可，不要过多。

## Phase 5：差距分析（输出 Part 5）

### 5a. 项目深挖清单

对面试中讨论到的每个项目：

1. 读取该项目的完整文档集（从 project-registry.md 取路径列表）
2. 对照面试中答得不好的问题，检查：
   - 项目文档中已覆盖 → 标注"你的文档里有，去复习：{文件名}"
   - 项目文档中未覆盖 → 标注"文档未覆盖，你需要自己想清楚并补充"
3. 输出为问题清单格式，每条是一个用户必须能用自己的话回答的问题

### 5b. 知识盲区清单

对面试中暴露的每个知识缺失：

| 概念 | 一句话解释 | 为什么你需要懂 | Obsidian参考 |
|------|----------|-------------|-------------|

按知识域分组（Agent / 评测 / PM方法论 / 其他）。

### 5c. 岗位匹配映射

| JD要求 | 匹配度 | 你的经验证据 | 面试中的表现 | 下次怎么讲 |
|--------|--------|------------|------------|----------|

末尾总结核心差距（按紧急度排序的3个待补项）。

### 5d. 本周训练任务（每条配参考答案）

基于本场面试暴露的问题，给出3-5个**本周内可执行的训练任务**。每条训练任务必须包含：

1. **任务描述** — 要训练什么具体能力（如"能用自己的话讲清3种Agent规划策略的trade-off"）
2. **参考答案** — 直接给出senior级标准回答内容（不是话术，是结构示范+关键论据+数据支撑）。基于用户真实项目和Obsidian知识库构建，**禁止编造**。如果某个数据/细节用户文档中没有，用 `⚠️ 此处需用户补充真实数据：xxx` 占位
3. **验收标准** — 用户怎么自检训练是否达标（如"能在2分钟内复述完整框架并能扛住2-3轮追问"）

输出格式：

```
#### 训练任务1：{任务描述}

**为什么练这个：** {基于本场面试哪个问题暴露出来的}

**参考答案（senior级）：**
{完整的标准回答，约150-300字，基于用户真实项目+知识库}
{答案来源标注：[项目文档：xxx.md] [知识库：xxx.md]}

**验收标准：**
- {可验证的具体行为，如"能默写完整框架"}
- {追问扛得住的标准，如"能回答'为什么不选B方案'"}
```

### 5e. 下次面试前自查项（每条配标准答案）

针对**简历上的每个数据/项目细节**，列出下次面试前必须能答出的关键问题。每条自查必须包含：

1. **自查问题** — 面试官最可能直接问的版本（如"跨镜头一致性0.65→0.88怎么定义？"）
2. **标准答案** — 可以直接说出口的完整回答（约100-200字）。**基于用户真实项目文档和知识库**，标注答案来源
3. **答案来源** — 这个答案来自哪份文档/哪条知识库笔记
4. **数据补充警告** — 如果答案需要的真实数据用户文档里没有，用 `⚠️ 此处需用户补充真实数据：xxx` 明确标注，提示用户必须先确认这个数据再使用

输出格式：

```
#### 自查问题1：{面试官可能怎么问}

**标准答案：**
{完整回答，可以直接说出口的版本，约100-200字}

**答案来源：** {项目文档路径 / 知识库笔记路径}

**⚠️ 待用户补充：** {如有数据空白，明确列出}
```

**核心约束：** 5d 和 5e 中的每个答案都必须能追溯到用户的真实项目或知识库。AI不能为了"看起来完整"而编造细节。文档中没有的，宁可空着标警告，也不能编造。

## Phase 6：提炼可复用话术 → 追加话术库

Part 1-5 生成后，从本场面试里提炼"换家公司还能用"的资产，追加进一份跨面试**常驻**的飞书「我的面试话术库」。这是持续累积的同一份文档，不是每场重建。

**库的完整结构、追加规范、去重规则、首次创建骨架，全部见 [`references/talk-track-bank.md`](references/talk-track-bank.md)。** 本节只讲"从本场提炼什么"。

### 收录标准：和你绑定，不是和这家公司绑定

判断一条内容要不要进库，只问一句：**换一家公司面试，这条还用得上吗？**

| 收（和你绑定，可复用） | 不收（和这家公司绑定） |
|---------------------|---------------------|
| 通用行为题（离职原因/职业规划/自我介绍/最大挑战/优缺点） | "为什么来我们公司"的具体版本 |
| 你的项目故事问答（你的项目不变，换哪家都讲） | 这家公司业务/产品细节的问答 |
| 你对常见 AI·PM 概念的标准解释（Agent/评测/RAG…） | JD 逐条匹配（那是 Part 5c 的事） |
| 面试官的通用提问套路 | 只在这家出现、不可迁移的细节追问 |

### Block 1：高频题 × 你的标准话术

对每条符合收录标准的问答，提炼一个条目：

1. **归一化题目** — 把本场问法抽象成标准题名（如「模型选型决策过程」），便于和库里已有题目匹配（同一意图 = 同一题，不看字面）
2. **面试官原话** — ⚠️ 关键：保留面试官**本场原始措辞**（verbatim），从**原始转录**里取（不是 Part 1 清洗后的版本）。只去纯噪音（嗯/啊），保留问法本身。标注来源 `{公司}·{日期}·{面试官角色}`
3. **你的标准答** — 可复用成品答案，code block 便于整段复制。来源分两类：
   - 项目题/概念题：基于项目文档+知识库构建，标来源
   - **行为题**：以你转录里的真实回答为底，打磨表达但**不改变事实**，标 `⚠️ 待你确认的真实版`——AI 不替你编离职原因这类只有你知道真相的内容
4. **标签** — 题型 / 适用面试官角色
5. **升级标注** — 若库里已有这道题、本场答案更好，标「升级版」并一句话说升了什么

### Block 2：面试官提问套路库

从转录里识别面试官的**提问手法**（不是问题内容，是"怎么问"）：

- 每条 = 套路名 + 触发场景 + 应对策略 + 出处（本场原话例子一句）
- 只提炼**本场新出现**的套路；通用的、库里已有的不重复写
- 本场没有值得记的新套路就省略，不硬凑

### 与现有 Part 的去重

库是"提炼层"，Part 1-5 是"本场分析层"。同一条内容**只在库里写完整成品答案**；复盘文档对应位置（Part 2/5e）继续负责"本场怎么答砸的、怎么训练"，不重抄成品。重叠时一句话互相指向即可。

### 本节产出两样东西

1. **话术库增量（delta）** — 合并进话术库文档的内容（Phase 8 发布）
2. **复盘文档的增量摘要段** — 一小段"本场新增 N 条已进话术库 + 链接 + 增量清单"，放复盘文档末尾（模板见 [`references/output-template.md`](references/output-template.md)）

## Phase 7：三重审核（发布前质量关卡）

所有 Part 1-5 内容和话术库增量生成完成后，**必须在发布前执行以下三轮审核**。这是硬性关卡，不能跳过。

### 审核1：可用性审核

**核心问题：用户能不能在真实面试中自然地说出来？**

逐条检查 Part 2-4 的内容：

| 检查项 | 不通过的表现 | 修正方式 |
|--------|------------|---------|
| 语言自然度 | 读起来像论文/教材，不像人在面试中会说的话 | 改成口语化表达，保留核心知识点但换成对话语气 |
| 知识密度 | 单个回答塞了太多概念，面试中根本说不完 | 拆分为核心回答 + 追问时再展开的备用内容 |
| 思维路径可操作性 | Step过于抽象（如"分析trade-off"），听完不知道具体想什么 | 替换为具体动作（如"想三个词：沉没成本、用户预期、失败分布"） |
| 数据可说出口 | 引用了用户其实记不住的精确数字 | 只保留用户能记住的关键数字（1-2个），其余改为量级描述 |

### 审核2：真实性审核

**核心问题：内容是否符合用户的真实经历？**

逐条交叉核对：

1. **数据核实** — Part 2中提到的所有数字（指标值、百分比、成本数据），必须与简历和项目文档中的数据一致。不一致的标注 `⚠️ 数据待确认：简历写X，这里写Y，请用户核实`
2. **经历核实** — "在你的项目中"部分描述的做法，必须在项目文档中有对应内容。项目文档中没有的，不能编造，必须标注 `⚠️ 文档中未找到此细节，请用户确认是否属实`
3. **角色边界** — 检查是否把别人做的事写成了用户做的。对比项目文档中用户的角色描述，确保归属正确
4. **时间线合理性** — 如果涉及时间（优化周期、上线时间），检查是否和用户的工作时间线吻合

### 审核3：知识库一致性审核

**核心问题：生成的"正确答案"是否和用户已有知识体系吻合？**

对 Part 2 中引用的每个知识框架/概念：

1. **回查知识库** — 重新读取 Phase 2 中引用的 Obsidian 文件，确认：
   - 生成内容中使用的术语和知识库中的一致（不能知识库写"Plan-and-Execute"，生成内容写"静态规划"）
   - 框架的分类方式和知识库一致（不能知识库分3类，生成内容分4类）
   - 没有引用用户知识库中不存在的框架（如果引用了，标注来源）

2. **标注知识来源** — 每个知识点标注来源类型：
   - `[来源：知识库]` — 来自用户 Obsidian vault 的内容
   - `[来源：项目文档]` — 来自用户项目文档的内容
   - `[来源：外部补充]` — 来自 WebSearch 的外部知识，用户vault中没有

3. **冲突检测** — 如果生成内容与知识库中的观点/数据存在矛盾，不静默覆盖，而是标注 `⚠️ 与知识库冲突：知识库中说X，此处建议Y，原因是Z，请用户判断`

### 话术库增量的额外检查

本场要追加进话术库的内容，额外过三关：

1. **原话忠实** — Block 1 的面试官原话必须和原始转录一致，不润色、不改写、不补全。宁可保留口语，也不替面试官"说得更顺"
2. **答案可溯源** — 项目题/概念题的标准答能指到文档或知识库；行为题答案已标 `⚠️ 待你确认的真实版`，没有 AI 编造的事实
3. **去重提示已加** — 和库里已有题疑似重复的，已标注 `⚠️ 可能重复，请确认是否合并`，没有静默合并、也没有重复建条目

### 审核输出

审核完成后，对整个文档标注审核状态：

- **全部通过** → 直接进入 Phase 8 发布
- **有警告项** → 在文档末尾汇总所有 `⚠️` 标注，作为"待用户确认事项"段落一并发布，并在返回结果中提醒用户检查
- **有严重问题**（如核心数据矛盾、编造经历）→ 暂停发布，先向用户报告问题，等用户确认后再修正发布

## Phase 8：组装飞书文档（复盘文档 + 追加话术库）

### Step 8.1：组装复盘文档

按以下结构组装 Lark-flavored Markdown：

参考 [`references/output-template.md`](references/output-template.md) 获取完整模板。

核心结构：
```
文档标题: "面试复盘 - {公司名} - {YYYY-MM-DD}"

0. 面试录音链接（标题正下方、概览之前的文档最顶部；没有则省略）
1. 复盘概览 callout（公司/岗位/日期/总体评估一句话）
2. Part 1: QA问答对（按主题H2分组）
3. Part 2: 正确答案（3-5个弱题的知识教学）
4. Part 3: 思维路径（代码块格式）
5. Part 4: 追问防御链
6. Part 5: 项目深挖清单 + 知识盲区清单 + 岗位匹配映射
7. 本场可复用话术增量摘要段（指向话术库，见 Phase 6 / output-template.md）
```

### Step 8.2：发布复盘文档

参考 [`references/feishu-publishing.md`](references/feishu-publishing.md) 获取发布配置。

```bash
# 创建文档
lark-cli docs +create \
  --title "面试复盘 - {公司} - {日期}" \
  --wiki-space my_library \
  --markdown "{第一段内容}" \
  --as user

# 长文档分段追加（每段控制在合理长度内）
lark-cli docs +update \
  --doc "{doc_id}" \
  --mode append \
  --markdown "{后续段落}" \
  --as user
```

### Step 8.3：追加话术库

追加话术库是**独立于复盘文档的另一份常驻文档**操作。完整规范（库结构、归一化、去重、初始骨架）见 [`references/talk-track-bank.md`](references/talk-track-bank.md)。

1. **定位库** — 读 `references/talk-track-bank.md` 顶部的 doc_token。
2. **首次（token 为空）** — 创建库文档，写入初始骨架，再用 Edit 工具把链接和 token 写回 `references/talk-track-bank.md`：
   ```bash
   lark-cli docs +create \
     --title "我的面试话术库" \
     --wiki-space my_library \
     --markdown "{初始骨架，见 talk-track-bank.md}" \
     --as user
   ```
3. **后续（已有 token）** — 用"读全库 → 内存合并 → 整体覆盖写回"，比中途插入可靠：
   ```bash
   # ① 完整读取现有库，确认读全（条目不能在合并后变少）
   lark-cli docs +fetch --doc "{bank_token}" --as user
   # ② 在内存里按追加规范合并本场增量：
   #    已有题→把本场原话追加进该题原话列表、答案有升级则更新
   #    新题→在对应题型分组下新建条目
   #    疑似重复→标 ⚠️ 不静默合并
   # ③ 用覆盖模式整体写回（具体 mode 用法查 lark-doc skill）
   lark-cli docs +update --doc "{bank_token}" --mode overwrite \
     --markdown "{合并后的完整库内容}" --as user
   ```
4. **安全护栏** — 覆盖前确认已完整读取旧库；合并后条目数只增不减；原话只增不删。

## 飞书Markdown注意事项

- **callout内只能放**：文本、列表、代码块
- **callout内不能放**：表格、分割线（---）、标题
- 表格放在callout外面
- 代码块用 ` ```plaintext {wrap} ``` ` 格式展示实际回答和思维路径
- 诊断用callout色彩区分：`light-green`=好 / `light-orange`=待改进 / `light-red`=关键失分

## 特别注意

1. **转录质量参差不齐** — 钉钉转录对技术术语识别差（Agent/Prompt/RAG等可能被错误转录），处理时做最佳努力修正
2. **上下文窗口管理** — 转录+项目文档+知识库可能很大。Phase 2只加载与弱题相关的文件，不一次性加载全部
3. **新项目不在注册表中** — 如果用户讨论了注册表外的项目，Phase 2-4仍可基于转录内容工作，Phase 5的项目深挖清单标注"无项目文档可参考"
4. **没有JD** — 跳过Part 5c岗位匹配映射，其余部分正常执行
5. **不编造经历** — 正确答案教学是教知识，不是替用户编造项目经历。如果用户的项目中确实没做过某件事，教用户怎么诚实地回答"这部分我没有直接负责，但我了解..."
6. **原话 vs 清洗，两套并存不冲突** — Part 1 的问答清洗口语是为了**可读**；话术库 Block 1 的面试官原话必须从**原始转录**取、保留措辞，是为了让用户对着**真实问法**演练复用。一个求可读、一个求真实问法，各取所需，不要用清洗版去填话术库的原话字段。
