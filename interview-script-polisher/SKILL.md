# Interview Script Polisher — AI PM 面试项目全流程准备

将产品经理的项目经历，从"执行者汇报"升级为"决策者的全流程产品思维展示"。

核心理念：面试官想听到的不是"你做了什么"，而是"你怎么发现问题、怎么判断、怎么设计、怎么验证"——覆盖产品全生命周期。

---

## 与其他 Skill 的关系

| Skill | 输入 | 核心职责 |
|-------|------|---------|
| `interview-story-writer` | 一个项目（从零开始） | **创建** STAR故事 + 简历描述 + 模块文档 |
| `interview-script-polisher`（本skill） | 已有项目材料/PRD | **深化** 为全流程面试展示材料 + 图表 + 追问预案 |
| `interview-reviewer` | 面试录音转录 | **复盘** 面试表现、补漏 |

本 skill 拿着已有项目（PRD、文档、口述）做全维度深化。如果用户还没有任何材料，先用 `interview-story-writer`。

## 审核原则（贯穿全流程）

**所有输出内容必须经过审核，符合现实情况，符合产品实际。**

### 三层信息标注

每段输出内容必须标注来源：
- 🟢 **来自材料**：直接从用户提供的 PRD/文档中提取
- 🟡 **公开验证**：通过 web search 查证的行业/竞品/市场信息
- 🔴 **待用户确认**：基于上下文推断，需要用户验证

### 审核检验标准

对每段输出问三个问题：
1. **这段内容有依据吗？** → 没有依据的删除或标注🔴
2. **面试官追问细节，候选人能答上来吗？** → 答不上来的标注风险 + 附安全话术
3. **和其他模块矛盾吗？** → 矛盾的修正

### 不通过审核的处理

- **无依据内容** → 删除，或降级为"建议补充"提问
- **推断性内容** → 标注🔴，转为确认问题
- **风险内容** → 保留但标注⚠️，附安全话术

## Step 2: 全流程梳理

按产品全生命周期整理项目，每个环节都用"决策者视角"而非"执行者视角"。

### 2a. 需求背景与市场判断

回答三个问题：
1. **这个产品/功能在公司的战略中处于什么位置？**
2. **为什么是这个时间点做？**（行业窗口/技术突破/竞争压力）
3. **做好了对业务意味着什么？**

用 web_search 搜索产品的公开信息，补全战略背景。🟡标注。

### 2b. 用户研究与分层

**核心原则：按使用动机分层，不按人口统计学分层。**

分层模板：

| 用户类型 | 典型场景 | 底层动机 | 核心期望 | 容忍底线 | 可观测行为 | 度量指标 |
|---------|---------|---------|---------|---------|-----------|---------|
| {动机}型用户 | ... | ... | ... | ... | ... | ... |

**关键要求**：用户分层必须贯穿后续所有产品决策。如果分层做完后面没有引用，说明分层没有决策价值。

详细方法论见 → references/user-segmentation.md

### 2c. 竞品分析与差异化

- 识别 2-3 个直接竞品 + 1-2 个间接竞品
- 用 web_search 验证竞品信息 🟡
- 每个竞品对比：方案差异 / 能力差距 / 体验差距
- 提炼差异化论点（一句话面试表达）
- 竞品壁垒判断：壁垒能维持多久？

### 2d. 产品定义与架构设计

三层架构梳理：
1. **宏观**：产品在整个 App/公司中的位置和战略角色
2. **中观**：功能模块全景 + 用户路径 + 功能优先级
3. **微观**：技术产品分层架构 + 核心数据流

关键架构决策：列出3个以上"为什么这么设计而不那么设计"的取舍点。

### 2e. 交互设计与核心流程

- 梳理核心用户路径（从进入到完成的完整链路）
- 标注关键决策点（用户在这里会做什么选择）
- 标注系统行为（系统在每个节点做了什么）
- 异常情况处理（出错/中断/边界case怎么办）

### 2f. AI能力设计（仅AI产品）

- **模型选型**：选了什么模型、为什么选它、对比过哪些备选
- **能力边界**：模型能做什么、不能做什么、边界case怎么处理
- **Prompt策略**：系统提示词的设计思路（不需要贴全文，讲思路）
- **Agent架构**（如有）：单Agent vs 多Agent、编排模式、上下文管理
- **评测体系**：评测维度、评测方法（人工/自动/对比）、评测集设计
- **迭代闭环**：评测结果如何驱动产品迭代

详细模板见 → references/ai-pm-competency.md

### 2g. 数据体系

**指标框架**：
```
北极星指标：{一个核心指标}
  ├── 支撑指标1：{指标名} — 衡量{什么}
  ├── 支撑指标2：{指标名} — 衡量{什么}
  └── 支撑指标3：{指标名} — 衡量{什么}
反向指标：{防止过度优化的制衡指标}
```

**埋点设计思路**（面试中讲"我怎么做数据驱动"的素材）：
- 关键行为 → 追踪什么事件 → 回答什么分析问题 → 驱动什么产品决策

### 2h. 结果验证与迭代

- 每个结果回扣开头的洞察/假设，形成"洞察→决策→验证"闭环
- 不只列数据，讲"验证了什么判断"
- 踩坑经历：遇到了什么问题、怎么解决的
- 方法论沉淀：这个项目让你沉淀下来的可复用思维框架

## Step 3: 生成展示图表

面试时，图表是最直观的展示方式。为项目生成以下图表（按需选择）：

### 必选图表

1. **产品架构图** — 展示产品的整体结构和模块关系
2. **核心用户流程图** — 展示主要用户路径和关键决策点

### 可选图表（根据项目类型）

3. **指标体系图** — 北极星 + 支撑指标树状结构
4. **评测维度图** — AI产品的评测维度和方法（仅AI产品）
5. **竞品对比矩阵** — 功能/能力对比表
6. **分期规划图** — 产品迭代路线图

### 图表生成方式

优先使用飞书画板（`lark-whiteboard` skill），嵌入飞书文档中。
备选：Mermaid 语法生成，以代码块形式嵌入文档。

图表原则：
- **简洁**：3-7个核心节点，不追求面面俱到
- **有决策点**：标注"为什么这么设计"的关键节点
- **面试友好**：30秒能讲清楚的粒度

详细模板见 → references/interview-diagrams-guide.md

## Step 4: 构建面试叙事（STAR+ 口语化）

### 4a. 叙事锚点选取

从 Step 2 的全流程材料中，选取 **3-5个最有力的叙事锚点**：

选取标准：
- 候选人亲手做的（不是团队其他人的贡献）
- 有数据支撑、经得起追问
- 展示了判断力和取舍能力
- 和目标岗位的核心能力匹配

### 4b. STAR+ 结构（升级版）

```
S — Situation
  ├── 战略背景（1-2句项目在公司中的位置）
  ├── 用户分析精华（最关键的用户洞察）
  └── 核心矛盾（什么问题驱动了这个项目）

T — Task
  ├── 你的角色和职责范围
  └── 你的核心判断（先做什么、为什么）

A — Action（围绕3-5个叙事锚点展开）
  ├── 锚点1：{环节名} — 核心决策 + 怎么判断 + 结果
  ├── 锚点2：...
  └── 锚点3：...

R — Result
  ├── 核心数据（回扣S中的假设）
  └── 验证了什么判断

D — Discussion
  ├── 踩坑经历（体现真实感）
  └── 方法论沉淀（可复用的思维框架）
```

### 4c. 时长控制

准备两个版本：
- **精简版（3-5分钟）**：S(30s) + T(30s) + A(2-3min, 3个锚点) + R(30s)
- **展开版（8-10分钟）**：全部 STAR+D，5个锚点，附图表讲解

### 4d. 口语化改造

在以下位置加入口语化表达：

| 位置 | 加入什么 | 示例 |
|------|---------|------|
| 每个阶段开头 | 一句话核心决策 | "这个阶段的核心思路是：先止血再优化" |
| 关键决策前 | 思考衔接语 | "那我是怎么判断的呢？""我们为什么选这个方向？" |
| 推动过程 | 阻力和应对 | "但说实话推动这件事不容易——" |
| 发现问题时 | 探索感 | "有一个发现出乎我意料——" |
| 架构决策时 | 方案对比 | "其实一开始考虑了三个方案..." |
| 讲指标时 | 数据叙事 | "有一个指标的变化让我注意到了问题..." |

详细口语化模式见 → references/conversational-patterns.md

口语化底线：
- 不能变成"聊天"——仍需有结构和逻辑线
- 关键结论仍用精炼、有力的表述
- 读出声念一遍：像"跟信任的同事复盘项目"→刚刚好

## Step 5: 审核验证

### 5a. 信息来源审计

逐段检查所有输出内容的信息来源标注（🟢/🟡/🔴），确保：
- 没有无标注的内容
- 🔴内容已转化为向用户的确认问题
- 🟡内容有 web_search 查证记录

### 5b. 面试现实感检验

对每个叙事锚点，模拟面试官追问：
- "这个数据从哪来？" → 候选人能说清数据源和口径吗？
- "这真的是你做的吗？" → 候选人能描述具体的执行细节吗？
- "为什么不做XX？" → 候选人能讲清取舍逻辑吗？
- "如果重来你会怎么做？" → 候选人有反思和复盘吗？

无法通过检验的内容 → 标注⚠️ + 设计安全话术

### 5c. 一致性检查

- 用户分层是否在后续决策中被引用
- 竞品分析的结论是否和差异化定位一致
- 指标体系是否对应了用户需求
- 时间线是否自洽
- 数据口径是否前后一致

### 5d. 审核输出

```
## 审核报告

### 信息来源分布
🟢 来自材料：XX%
🟡 公开验证：XX%
🔴 待确认：XX%

### 面试追问风险项
{列出无法通过追问检验的内容 + 安全话术}

### 一致性问题
{列出发现的矛盾点 + 修正建议}
```

## Step 6: 追问预案（独立子页面）

### 6a. 按维度分类

为每个覆盖的维度整理高频追问：

**用户分析追问**
- Q: 你怎么知道用户需要这个？
- Q: 用户调研的样本量是多少？
- Q: 你的分层依据是什么数据？

**竞品追问**
- Q: 竞品X也做了类似的，你的优势是什么？
- Q: 你的壁垒能维持多久？

**架构追问**
- Q: 为什么选这个架构/技术方案？
- Q: 考虑过其他方案吗？为什么不选？

**AI能力追问**（AI产品）
- Q: 模型准确率是多少？怎么评测的？
- Q: 遇到badcase怎么处理？
- Q: Prompt是怎么设计的？

**数据指标追问**
- Q: 你的核心指标是什么？为什么选这个？
- Q: 数据提升的归因是什么？怎么排除其他因素？

**协作追问**
- Q: 和研发有分歧怎么办？
- Q: 怎么推动跨团队协作？

### 6b. 连锁追问链

面试官深挖时的追问路径，每个节点都有预案：

```
Q1: 这个功能的用户是谁？
  → A1: {回答}
  → Q2: 你怎么做的用户调研？
    → A2: {回答}
    → Q3: 样本量多少？有没有代表性问题？
      → A3: {回答 / 安全话术}
```

### 6c. 风险追问与安全话术

对每个🔴和⚠️标注的内容：

```
**Q: {面试官可能的追问}**
> 风险等级：🔴高 / 🟡中
> 推荐回答：{尽量回答的部分}
> 安全话术：{细节模糊但展示思维框架的兜底表达}
> 引导方向：{如何把话题引回强项}
```

### 6d. 追问引导策略

当面试官追问到不熟悉的区域时：
1. 诚实承认："这部分具体的数字我需要确认一下"
2. 展示框架："但我的思考逻辑是这样的..."
3. 引导回强项："这让我想到我在XX方面做得比较深..."

输出前逐项检查：

- [ ] 每个环节有一句话核心决策概括
- [ ] 用户分层贯穿了后续产品决策
- [ ] 至少使用了一个产品方法论且有依据
- [ ] Result回扣了Situation中的洞察
- [ ] 口语化衔接自然，不像背稿
- [ ] 所有时间线自洽
- [ ] 数据口径清晰
- [ ] 追问预案覆盖至少10个高风险问题
- [ ] 没有使用不能解释的术语
- [ ] 所有内容有信息来源标注（🟢/🟡/🔴）
- [ ] 图表简洁、30秒能讲清楚
- [ ] 精简版叙事控制在5分钟以内
- [ ] 飞书文档成功创建并返回链接
