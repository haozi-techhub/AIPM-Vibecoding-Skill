# 竞品分析工作流 Skill

产品经理在评估新产品方向时，按 **8步顺序** 完成从"发现机会"到"立项报告"的完整调研链路。每步必须用 browse 工具从真实网页获取数据，禁止杜撰，无法获取的内容明确告知用户。

---

## AI 产品分析框架（前置知识）

分析一个 AI 产品，可从下至上拆解为**四层架构**：

| 层级 | 关注点 | 举例 |
|------|--------|------|
| **用户层** | 用户可直接操作的交互界面与功能 | 对话框、功能按钮（PPT生成/深度研究）、输出展示 |
| **技术层** | 支撑功能的 AI 技术应用 | 核心技术（RAG/Long CoT/Prompt/Agent）、其他技术（图像识别/OCR/ASR/TTS/多语种对齐） |
| **模型层** | 产品使用的核心模型 | 自研模型、开源模型微调、第三方 API 调用 |
| **基础设施层** | 模型训练与部署的底层支撑 | 算法架构（MoE/RLHF）、数据库（向量DB）、算力资源 |

**不同受众关注不同层级**：

| 报告受众 | 重点维度 | 分析深度 |
|---------|---------|---------|
| 老板/领导 | 市场定位 + 产品功能 + 用户体验 + 技术内核 | 全面，战略视角 |
| 产品团队 | 市场定位 + 产品功能 + 用户体验 | 聚焦迭代方向 |
| 研发团队 | 模型技术 + 产品功能 | 聚焦技术路线 |
| 运营团队 | 用户体验 + 产品功能 | 聚焦转化与活跃 |

**AI 产品阶段与评测侧重**：
- **L1 对话型**：评测对话能力、响应质量
- **L2 推理型**：评测推理深度、逻辑严密性
- **L3 Agent 型**：评测任务完成度、自主规划能力

## 使用前置条件

用户需提供：
1. **产品方向**：具体产品名称/功能描述
2. **目标用户假设**：3~5类用户群体
3. **核心场景假设**：用户在什么情况下使用
4. **成功标准**：什么情况下"值得做"（如市场增速、竞争格局、未满足痛点）

如果用户没有提供完整信息，主动询问补齐。

---

## Step 1：决策问题拆解

**触发词**：用户提供产品方向和成功标准后，立即执行

### 1.1 确认报告受众

先问用户：这份分析报告主要给谁看？根据受众自动调整后续步骤的侧重（参考上方「不同受众关注不同层级」表格）。

### 1.2 拆解决策子问题

```
分析目标：{产品方向}
决策类型：做/不做
报告受众：{老板/产品/研发/运营}

需要回答的子问题：
□ 市场层：规模多大？增速多少？生命周期？
□ 竞争层：直接竞品是谁？护城河高低？
□ 用户层：谁在用？为什么用？痛点是什么？
□ 功能层：竞品做了什么？没做什么？
□ 技术层：用什么模型？什么技术栈？性能如何？
□ 商业层：怎么赚钱？成本结构？
□ 差异化：我们能做别人做不了的是什么？
```

### 1.3 竞品选择方法论

从两个维度帮用户圈定竞品范围：

**维度一：关联性**
- **直接竞品**：产品目标、用户人群高度重叠（~90%），如抖音与快手、豆包与 Kimi
- **间接竞品**：解决类似需求但方式不同，有部分重叠，如 Midjourney 与 ChatGPT
- **潜在竞品**：产品完全不同，但争夺用户同一项有限资源（时间/金钱），如王者荣耀与抖音

**维度二：分析方向**
- **业务方向**：服务于相同商业目标，重点分析场景替代性，如飞书与钉钉
- **功能方向**：在某个垂直功能上对标，重点分析性能指标与体验细节，如 Cursor 与 Copilot
- **用户方向**：目标用户高度重叠，重点分析需求匹配度与迁移成本，如豆包与通义

输出：结构化问题清单 + 竞品选择方向，确认用户对齐后进入 Step 2。

## Step 2：竞品识别

**Browse 搜索渠道**（并行执行，提升效率）：

```
搜索平台：
- 36氪：搜索"{产品方向} 融资" "{产品方向} 创业"
- ProductHunt：搜索英文关键词
- 小红书：搜索"{产品方向}" "{核心场景}"
- IT桔子：搜索相关公司融资记录
- App Store / 安卓应用市场：搜索相关 App，查看下载量和评分
- 七麦数据：查看 App 排名与下载趋势
- 垂直社区：行业官网/论坛（根据产品方向补充）
```

**竞品分类输出**：

| 类别 | 定义 | 要求输出字段 |
|------|------|------------|
| 直接竞品 | 做一样的事，抢一样的用户 | 名称/官网/核心功能/目标用户/融资阶段/信息来源 |
| 间接竞品 | 解决类似需求但方式不同 | 同上 |
| 替代品 | 用户现在实际用什么解决 | 同上 |
| 潜在竞品 | 有能力进来但还没进的大公司 | 同上 |

**规则**：每条信息必须标注来源URL，无法获取的字段注明"待补充"，禁止推测。

## 执行规则（所有步骤通用）

1. **数据必须标来源**：每个数据点附 URL 或报告名称+年份
2. **禁止杜撰**：拿不到的内容明确说"无法获取，请用户补充"
3. **信息有冲突时**：列出多个来源，说明差异，不擅自取舍
4. **Browse 工具策略**：优先直接抓取官网/App Store，次选搜索引擎，两者都失败才让用户补充
5. **步骤可并行**：同一步骤内的多个竞品可并行抓取，提升效率
6. **发现重要信息立即标注**：关键发现用加粗或引用块突出显示
7. **AI 产品必做技术层**：如果判断分析对象是 AI 产品，Step 7 不可跳过
8. **报告结论前置**：每个步骤的输出开头先写 1-2 句核心结论，再展开细节
